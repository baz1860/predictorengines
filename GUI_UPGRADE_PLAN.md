# GUI + Dashboard Upgrade Plan

The desktop app (`app/`) and the offline `dashboard.html` both work, but they read as a thin first pass: flat cards, one accent colour, no charts in the app, no dark mode, static tables, and a dashboard that's a generated snapshot rather than something you can interact with. This plan makes the whole surface feel like a finished product — richer visually, denser with data, interactive, and with screens for things the engines already compute but the UI never shows.

Scope was confirmed with Barrie: **all four directions** (visual polish, more data & charts, interactivity, new capability screens), the **dashboard folds into the app as a live view**, and we **plan then build** in the same session.

## What exists today

**Desktop app** — PyWebView window over a FastAPI backend (`app/server.py`), static frontend in `app/web/` (`index.html`, `style.css`, `app.js`, vanilla JS, no build step, no libraries). Layout is a dark sidebar (engine list + Bankroll + Settings) and a light main area with per-engine tabs Predict / Simulate / Edge. The backend is cleanly generic: every engine is an adapter exposing `info()` and capability methods, dispatched through `/api/predict|simulate|edge|bankroll|settings`. Adding a screen means adding a route + a panel, not touching engine code.

**Offline dashboard** — `report.py` reads local CSVs and the suite ledger and writes a single self-contained `dashboard.html` (inline SVG, no JS). Cards: summary, bankroll curve, CLV trend, calibration, today's fixtures, bet queue, title-odds movers. It's a point-in-time artifact regenerated by the daily pipeline.

The data the UI *could* show but doesn't: bankroll history as a curve inside the app, calibration, CLV, fixtures, the bet queue, title-odds movement, and any drill-down on a prediction or an edge row.

## Design principles for the upgrade

1. **One design language across app and dashboard.** Shared token set (colour, spacing, radius, shadow, type scale) so the live dashboard and the engine screens look like one product. Today the app is light + blue, the dashboard is dark + green; they should converge.
2. **Stay dependency-light and offline.** The app ships as a packaged `.app` and the dashboard opens via `file://`. No CDN, no build step. Charts are hand-rolled SVG in a small reusable JS module (vendored locally if we ever want a library, never fetched at runtime). This keeps the "no rewrite, reuse the Python" promise of the original GUI plan.
3. **Backend stays generic.** New screens are new routes returning JSON; the adapter registry pattern is preserved. The dashboard becomes a route (`/api/dashboard`) backed by a small data module, not a special case.
4. **Don't break what works.** Existing markup IDs/classes that `app.js` binds to are preserved or migrated deliberately. Each phase is independently shippable and testable against the existing CLI outputs.

## Workstream 1 — Visual system

Replace the current CSS with a proper design system in `app/web/style.css`:

- **Tokens & theming.** A full CSS-variable palette (surface levels, borders, text tiers, semantic win/draw/loss/positive/negative, accent) with a **dark theme** as a `data-theme` attribute toggle and a light theme retained. Respect `prefers-color-scheme` on first load; let the user override from a header control. The dashboard's existing dark palette (`#0f1419` / `#161b22` / `#2fbf71`) informs the dark theme so the two surfaces match.
- **Depth & hierarchy.** Soft shadows and layered surfaces instead of flat 1px borders everywhere; clearer separation between toolbars, cards, and tables. Sticky table headers get a subtle elevation on scroll.
- **Typography.** A defined type scale (display / title / body / label / numeric), tabular-figure numerics everywhere numbers line up, tighter and more deliberate spacing.
- **Iconography.** A small inline-SVG icon set (sport glyphs for the sidebar, nav icons, status/trend arrows) — inlined, no icon font, no network.
- **Component polish.** Restyled buttons (primary/ghost/danger with hover/active/disabled states), inputs and selects with consistent focus rings, segmented controls for model/market toggles, pill/badge components for tags and edge signals, skeleton-loading and empty states, toast/inline confirmations instead of `confirm()` and bare notes.
- **Motion.** Tasteful, fast transitions (panel switches, result reveals, spinner→content) — nothing that delays interaction.

## Workstream 2 — Live in-app Dashboard (folding in `report.py`)

Add a suite-level **Home / Dashboard** view, the landing screen when the app opens.

- **Backend.** New `app/dashboard_data.py` computes a single JSON payload from the same local files `report.py` already reads (suite ledger, `predictions_worldcup_2026.csv`, `bet_queue.csv`, `tournament_odds.csv`, `data/title_history.csv`, calibration maps, `clv.py`). Exposed as `GET /api/dashboard`. `report.py` is refactored to consume the same module so the offline file and the live view never drift — one source of truth, two renderers.
- **Frontend.** A `panel-dashboard` rendered by a new charting module. Cards: headline KPIs (bankroll, net P&L, open at-risk, settled record, ROI, hit-rate) as stat tiles with trend arrows; **bankroll curve** (interactive line/area with hover tooltips); **CLV trend**; **calibration** plot; **today's/next fixtures** with the win/draw/loss bar inline; **live bet queue**; **title-odds movers** with ▲▼ deltas. Reload pulls fresh data; no regeneration step.

## Workstream 3 — Charts & richer data (`charts.js`)

A new `app/web/charts.js`: a tiny, dependency-free SVG chart kit reused by every screen.

- `lineChart` / `areaChart` (bankroll, CLV) with axis labels, baseline, gridlines, and hover tooltips.
- `sparkline` (inline trend in stat tiles and table rows).
- `barChart` / horizontal bars (title odds, simulate results, edge distribution).
- `scatter` / step (calibration).
- `probBar` extracted from the current predict markup into a reusable component.

These replace the current ad-hoc inline-SVG-in-Python and the single CSS prob-bar, and feed both the dashboard and the per-engine result panels (e.g. a scoreline heat strip, a win-probability donut on Predict).

## Workstream 4 — Interactivity

- **Tables become live:** client-side sort (click headers), filter/search box, and column show/hide on the wide tables (Simulate, Edge, Bankroll ledger). Edge and bankroll tables get sport/market filters.
- **Row drill-down:** click an edge row to expand model vs market probabilities and the Kelly math; click a fixture to open its full prediction without leaving the dashboard.
- **Persisted preferences:** theme, default engine, last-used model, and table sort/filters persisted via the backend settings store (the app already has `settings_store.py`) so choices survive restarts. *(Note: in-app persistence uses the settings store / disk, not browser localStorage.)*
- **Live bankroll refresh & inline settle:** settle results and reset with toast feedback and an animated bankroll update, no full-panel reload.

## Workstream 5 — New capability screens

Surface what the engines already produce:

- **Dashboard / Home** (Workstream 2) — the big one.
- **Bet history explorer** — full ledger view (today only "recently settled" shows): filter by sport/engine/market/date, P&L curve, ROI and hit-rate by market, CLV per bet. Built on the existing ledger.
- **Model comparison on Predict** — run the same fixture through multiple models (elo/dc/blend) side by side instead of one at a time.
- **Fixtures / schedule view** — upcoming predicted fixtures beyond just "today", with the edge picks attached.
- **Outrights / title-race screen** (soccer) — the tournament-odds table with movement history as a chart, not just the dashboard mini-card.

## Build order

Each phase is shippable and testable on its own.

1. **Design system foundation.** New `style.css` (tokens, dark/light, depth, type, components, icons) + a header theme toggle. Re-skin the existing screens with zero behavioural change — pure visual upgrade, verify every current screen still works. *(started this session)*
2. **Charting module.** `charts.js` with line/area/bar/sparkline/scatter + reusable prob-bar; unit-eyeball against known data. *(started this session)*
3. **Live Dashboard.** `dashboard_data.py` + `/api/dashboard` + the Home panel; refactor `report.py` onto the shared module so offline and live agree. *(started this session)*
4. **Interactivity layer.** Sortable/filterable tables, row drill-downs, toast/skeleton states, persisted prefs.
5. **New screens.** Bet-history explorer, model comparison, fixtures, outrights.
6. **Verification & polish.** Golden-output check (GUI numbers == CLI numbers for the same inputs), dark/light pass, empty/error/loading states on every panel, then a packaged-app smoke test.

## Risks / decisions

- **Offline charting vs a library.** Hand-rolled SVG keeps the bundle tiny and truly offline but costs build effort. If we later want richer interactions (zoom/pan, large series), we vendor a small library (e.g. uPlot) *locally* into `app/web/`, never via CDN. Starting hand-rolled.
- **`report.py` refactor.** Folding its data logic into `dashboard_data.py` must keep the offline file byte-for-byte useful for the daily pipeline. Mitigation: `report.py` keeps writing `dashboard.html`, just sourcing data from the shared module; verify the generated file before/after.
- **Markup migration.** `app.js` binds to specific IDs. The re-skin preserves those hooks (or migrates them in lockstep) so nothing silently breaks; verified screen-by-screen in Phase 1.
- **Theme persistence.** Uses the on-disk settings store, consistent with how API keys/defaults already persist — not browser storage.
- **Packaging weight.** Unchanged from the original plan; no new runtime deps added.
